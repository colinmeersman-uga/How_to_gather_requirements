# Requirements Gathering Checklist

Use this checklist to align requirements discovery with the RevOps Agile user-story process.

## Before the Meeting

- [ ] Review the request, ticket, or stakeholder note.
- [ ] Identify likely user roles and stakeholders.
- [ ] Gather available examples, screenshots, transcripts, reports, forms, or files.
- [ ] Identify likely systems involved.
- [ ] Prepare open-ended questions.
- [ ] Prepare to capture user stories, acceptance criteria, open questions, and follow-up owners.

## During the Meeting: Discovery

### 1. Identify User Roles

- [ ] Who is the end user?
- [ ] Who performs the work today?
- [ ] Who receives the output?
- [ ] Who validates success?
- [ ] Who owns the process?

### 2. Understand User Needs

- [ ] What happens today?
- [ ] What problem are users facing?
- [ ] What is manual, slow, risky, duplicated, or unclear?
- [ ] What business value should the solution deliver?
- [ ] What does success look like?

### 3. Map Workflow and Systems

- [ ] What starts the process?
- [ ] What systems create, store, update, or consume the data?
- [ ] Which system is the source of truth?
- [ ] Are there access or permissions issues?
- [ ] What examples can stakeholders provide?

## During the Meeting: Story Creation

### 4. Write User Stories

- [ ] Stories use: As a [type of user], I want [an action] so that [a benefit].
- [ ] Each story has a clear user.
- [ ] Each story has clear business value.
- [ ] Stories avoid unnecessary technical jargon.

### 5. Define Acceptance Criteria

- [ ] Criteria are clear and testable.
- [ ] Criteria cover the happy path.
- [ ] Criteria cover relevant failure paths.
- [ ] Criteria define what should be logged or visible.
- [ ] Criteria can be validated by stakeholders and testers.

### 6. Identify Exceptions

- [ ] What happens if required data is missing?
- [ ] What happens if no match is found?
- [ ] What happens if multiple matches are found?
- [ ] What happens if the destination does not exist?
- [ ] What happens if the item is incomplete, invalid, or duplicated?
- [ ] What happens if the item was already processed?

### 7. Capture Audit and Notifications

- [ ] What should be logged?
- [ ] Who needs visibility?
- [ ] Should anyone be notified?
- [ ] Should a related record be updated?
- [ ] How will the team troubleshoot issues?

## After the Meeting: Backlog Readiness

### 8. Estimate Story Points

- [ ] Development team has enough context to estimate.
- [ ] Estimate includes system complexity, data risk, testing, and dependencies.
- [ ] Unknowns are documented.

### 9. Prioritize Stories

- [ ] Stories are added to the backlog.
- [ ] Priority reflects value, dependencies, and urgency.
- [ ] MVP scope is identified.
- [ ] Lower-priority enhancements are separated from first-version requirements.

### 10. Refine, Plan, Develop, Test, and Demo

- [ ] Stories are reviewed in refinement.
- [ ] Acceptance criteria are updated based on feedback.
- [ ] Sprint candidates are selected based on priority and capacity.
- [ ] Test data and validation owners are identified.
- [ ] Completed work is reviewed or demoed with stakeholders.
- [ ] Feedback becomes new or updated backlog items.

## Quality Gate

Before a story is ready for development, confirm:

- [ ] User role is clear.
- [ ] User value is clear.
- [ ] Acceptance criteria are testable.
- [ ] Source of truth is identified.
- [ ] Dependencies are known.
- [ ] Edge cases are captured or intentionally deferred.
- [ ] Priority is assigned.
- [ ] Estimate is captured or ready for team estimation.
- [ ] Open questions have owners.
